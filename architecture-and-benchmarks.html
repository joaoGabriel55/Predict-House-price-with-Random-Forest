House Price Predictor

Architecture & Performance Benchmarks

Random Forest + LLM Hybrid System

Overview

A machine learning web application that predicts house prices using a Random Forest Regressor, built with Ruby, Sinatra, and Rumale. The application features a unique hybrid approach combining traditional ML with LLM-powered natural language understanding.

Key Innovation: The system offers two prediction modes:
  • Traditional Mode: Direct form-based input for structured data
  • AI Text Mode: Natural language descriptions processed by Claude Opus 4.6, which extracts parameters and calls the Random Forest model as a tool

System Architecture

Project Structure

. ├── controllers/ │ └── predictions_controller.rb # Handles prediction requests ├── models/ │ ├── house_predictor.rb # Singleton ML model wrapper │ └── house_predictor_llm.rb # LLM Tool for RF predictions ├── views/ │ ├── index.erb # Traditional form interface │ ├── result.erb # Traditional results page │ ├── llm_index.erb # AI text mode interface │ └── llm_result.erb # AI text mode results ├── public/ │ └── style.css # Application styles ├── server.rb # Sinatra app entry point ├── benchmark.rb # Performance benchmarks ├── house_model.dat # Serialized trained model (50MB) ├── houses.csv # Training dataset (50,050 rows) └── generate_samples.py # Data generation script

MVC Architecture Pattern

M
Models - HousePredictor (singleton) loads the trained Random Forest model once at startup. HousePredictorLLM wraps it as a RubyLLM Tool for LLM integration.
V
Views - ERB templates render the user interface for both traditional form-based input and AI text mode, displaying results with extracted parameters.
C
Controller - PredictionsController handles routing, parameter validation, and orchestrates model predictions for both prediction modes.

Data Flow Diagrams

1. Traditional Prediction Pipeline

1
User submits form with area, rooms, bathrooms, age
2
POST to /predict → PredictionsController extracts params
3
Controller calls HousePredictor.instance.predict()
4
Features converted to Numo::DFloat array: [[area, rooms, bathrooms, age]]
5

2. AI Text Mode Pipeline (Tool-Based)

1
User enters natural language description: "A spacious 250m² house with 5 bedrooms, 3 bathrooms, 10 years old"
2
POST to /llm/predict → PredictionsController.predict_with_llm()
3
RubyLLM context created with HousePredictorLLM tool registered (choice: required)
4
Claude Opus 4.6 analyzes text and extracts structured parameters
5
LLM calls HousePredictorLLM.execute(area: 250, rooms: 5, bathrooms: 3, age: 10)
6
Tool internally calls HousePredictor.instance.predict() → Random Forest prediction
7
Tool returns result to LLM with extracted params and predicted price
8
Result rendered in llm_result.erb showing extraction + prediction (avg latency: 4,040ms)

3. Model Training Pipeline

1
generate_samples.py creates synthetic data using formula: price ≈ 2.2×area + 20×rooms + 15×bathrooms − 3×age ± 5% noise
2
Data saved to houses.csv (50,050 samples)
3
save_model.rb loads CSV and splits data into train/test sets
4
Random Forest trained: n_estimators: 100, max_depth: nil, random_seed: 42
5
Model serialized with Marshal.dump to house_model.dat
6
HousePredictor singleton loads model at app startup (loaded once, reused for all requests)

Key Findings & Recommendations

Hybrid Architecture is the Optimal Solution

The tool-based approach (LLM + RF Tool) combines the best of both worlds:

  • LLM Strength: Natural language understanding, flexible input formats, better UX
  • RF Strength: Accurate, fast, deterministic predictions on structured data
  • Result: Users can describe houses naturally while maintaining ML precision

When to Use Each Approach

Use Random Forest When:

  • Production accuracy is critical (MAE: $8.2K vs $155K)
  • High throughput needed (2,222 vs 0.25 predictions/sec)
  • Deterministic results required (0 variance)
  • Cost optimization matters ($0 vs $0.05/prediction)
  • Input data is already structured
  • Real-time predictions required

Use LLM (Tool-Based) When:

  • User-facing interface needs natural language
  • Input format varies widely (unstructured text)
  • UX demands conversational interactions
  • Building chatbots or voice assistants
  • Processing listing descriptions or user queries
  • Latency tolerance is high (>4 seconds acceptable)

Recommended Production Strategy

Dual-Mode Implementation:

  • API/Backend: Direct Random Forest for maximum speed and accuracy
  • User Interface: Hybrid tool-based approach for better UX
  • Cost Control: Cache LLM extractions for similar queries
  • Fallback: If LLM extraction fails, provide traditional form

100% extraction accuracy proves the LLM can reliably parse natural language into structured parameters that feed the RF model.

Architecture Strengths

  • Singleton Pattern: Model loaded once at startup, reused across all requests (efficient memory usage)
  • Tool-Based Design: LLM autonomously decides when to call the RF tool, enabling intelligent workflows
  • Clear Separation: MVC architecture keeps concerns isolated (business logic, presentation, routing)
  • Type Safety: Numo::DFloat ensures correct numerical types for ML predictions
  • Error Handling: Graceful fallbacks in both traditional and AI text modes

Technical Implementation Details

Random Forest Configuration

Model: Rumale::Ensemble::RandomForestRegressor Parameters: - n_estimators: 100 # Number of decision trees - max_depth: nil # Trees grow until pure leaves - random_seed: 42 # Reproducible results - n_jobs: -1 # Use all CPU cores Training Data: - Total samples: 50,050 - Features: [area, rooms, bathrooms, age] - Target: price (in thousands) - Train/Test split: 80/20 Performance: - MAE: $8,210 - RMSE: $10,800 - Prediction time: 0.45 ms

LLM Tool Integration

Tool: HousePredictorLLM (extends RubyLLM::Tool) Model: Claude Opus 4.6 (via OpenRouter API) Tool Schema: execute(area:, rooms:, bathrooms:, age:) → { area, rooms, bathrooms, age, predicted_price } Configuration: - choice: :required # Forces LLM to call the tool - callback: after_tool_result # Captures tool output - context: OpenRouter API key from .env Workflow: 1. User describes house in natural language 2. LLM receives prompt with tool registration 3. LLM autonomously extracts parameters 4. LLM calls HousePredictorLLM.execute(...) 5. Tool internally calls HousePredictor.instance.predict() 6. Result returned to LLM, then displayed to user

Data Generation Formula

Python Script: generate_samples.py Formula: price = (2.2 × area) + (20 × rooms) + (15 × bathrooms) - (3 × age) price += random_noise(-5% to +5%) Ranges: - area: 40-500 square meters - rooms: 1-10 - bathrooms: 1-6 - age: 0-50 years - price: ~$50K - $1,500K Output: houses.csv (50,050 synthetic samples)

Conclusion

This project demonstrates a hybrid ML + LLM architecture that leverages the strengths of both approaches:

  • Random Forest provides accurate, fast, deterministic predictions (MAE: $8.2K, 0.45ms latency)
  • Claude Opus 4.6 enables natural language understanding with 100% extraction accuracy
  • Tool-based integration allows the LLM to intelligently call the RF model as needed
  • Dual-mode interface serves both technical users (form) and general users (natural language)

Final Recommendation

For production house price prediction: Use Random Forest directly for maximum accuracy, speed, and cost-efficiency.

For user-facing applications: Implement the hybrid tool-based approach to provide intuitive natural language input while maintaining RF's prediction accuracy.

The 19x accuracy advantage and 8,985x speed advantage of Random Forest make it the clear choice for the prediction engine itself. The LLM serves as an intelligent interface layer, not a replacement.
